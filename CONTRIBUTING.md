# Contributing to S-Blog

## 🎯 Project Vision & Boundaries

**Vision:** S-Blog aims to explore the absolute extreme limits of pure static blogs—specifically focusing on Single Page Application (SPA) architecture, strict SEO-friendliness, and highly simplified configuration. Simultaneously, this project serves as an experiment to push the boundaries of 100% AI-driven programming.

**Boundaries:** We exercise strict feature restraint. S-Blog will **never** include custom server-side logic (backend) and will **never** evolve into a centralized platform. While third-party client-side services (e.g., comment plugins) are permitted for necessary dynamic features, their integration must be kept strictly minimal.

---

## ⚠️ AI-Only Development & Review

- **Zero Manual Coding:** All code in this project must be generated by AI. Manual coding is strictly prohibited. Use agentic tools (e.g., Claude Code, Kiro, CodeX) to generate your contributions.
- **Pre-PR AI Validation:** Code review happens *before* submitting a PR. You must ensure your AI agent has completed rigorous self-checking and cross-validation locally to catch logical errors and hallucinations prior to opening a Pull Request.

---

## 💬 Language Policy

While the official language of this repository is English, you are warmly welcome to submit issues, pull requests, or general suggestions in **any language**. The only requirement is that your descriptions, logic, and reproduction steps are expressed clearly.

---

## Getting Started

1. Fork the repository
2. Clone your fork locally
3. Install dependencies: `npm install`
4. Create a branch from `master`

## Issues

- Search existing issues before opening a new one
- Use a clear, descriptive title
- Include steps to reproduce for bugs
- Specify expected vs actual behavior
- Add relevant labels (`bug`, `feature`, `docs`, etc.)

## Pull Request Flow

1. Create a feature or fix branch from `master`
2. Make your changes (AI-generated only)
3. Test locally with `npm run dev` and `npm run build`
4. **Cross-validate:** If using AI for self-review, complete multiple rounds of validation to catch potential hallucinations or logical loops.
5. Push your branch and open a PR targeting `feature/npm-publisher`
6. Ensure CI passes and request a review
7. Once approved, the maintainer will merge, publish, and sync to `master`

## Branching Model

```
master (stable)
 ├── feature/your-feature    ← branch from here
 └── fix/your-bugfix         ← branch from here
          ↓ PR
feature/npm-publisher (integration & publishing)
          ↓ after publish
master
```

- **`master`** — stable, production-ready code. Create your branches from here.
- **`feature/npm-publisher`** — integration branch where changes are tested and published. **All PRs should target this branch.**
- **`feature/*` / `fix/*`** — short-lived branches for individual changes.

## Commit Messages

Use clear, concise commit messages. Prefix with type:

- `feat:` new feature
- `fix:` bug fix
- `docs:` documentation changes
- `chore:` maintenance tasks

## Code Style

Follow existing project conventions. The AI tools should maintain consistency automatically. Do not introduce new libraries or tools without discussion in an issue first.
