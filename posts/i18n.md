## Overview

spage provides built-in internationalization (i18n) at two levels:

1. **UI language** — Navigation, buttons, and system text support English, Chinese, and Japanese
2. **Post content** — The same post can have multiple language versions

---

## UI Language Switching

> 💡 **Try it now:** Click the language switcher next to the site title at the top of this page (中 | En | 日) to preview the UI language switching in action.

A language switcher (**中 | En | 日**) is available in the header. Clicking it switches the entire site UI.

The system detects the user's language preference from `localStorage` (previous choice) or browser settings.

## Multi-language Posts

Use filename suffixes to create localized versions of a post:

```
posts/
├── About.md          # Default version
├── About.zh-CN.md    # Chinese version
└── About.ja.md       # Japanese version
```

All files with the same slug are recognized as language variants of one post. The build engine automatically tracks available languages in the manifest.

### Fallback Behavior

When a user's language has no matching post version:
- The default version is displayed
- A notice bar informs the user

## Configuration

The default `language` is configured via the language field in in `config.json`:

```json
{
  "language": "en"
}
```

This field sets the site's default language and affects the following behaviors:

1. Article List Display — Prioritizes localized titles and summaries if available.
2. Fallback Behavior — Users browsing in a language that matches the site's default won't see fallback notices (as they are already viewing the "original" content).

If this field is not set, the system defaults to `en`.

## For Developers

### Built with

spage's i18n is implemented based on  [i18next](https://www.i18next.com/) + [react-i18next](https://react.i18next.com/) :

- **Translation Resources** — Inlined in the `i18n.ts` file of `@s-page/core`
- **Language Detection** — Uses `i18next-browser-languagedetector`, prioritizing localStorage, then navigator
- **Multilingual Content** — During manifest generation, the build engine scans filename suffixes and outputs the `availableLanguages` and `localizedMeta` fields


### Manifest Data Structure

In the built `manifest.json`, multi-language articles will contain the following additional fields:

```json
{
  "slug": "About",
  "title": "关于本博客系统",
  "availableLanguages": ["en"],
  "localizedMeta": {
    "en": {
      "title": "About This Blog System",
      "summary": "An introduction to the blog system behind this site — how it works, how to use it, and how to deploy."
    }
  }
}
```

### Extending Translations

If you have forked this project and wish to modify the UI translation strings, the translation resources are located at:

```
packages/core/src/i18n.ts
```

Translations for each language are organized as nested objects, containing namespaces such as `nav`, `common`, `albums`, and `post`.
